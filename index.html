WS0: Problem Description만으로 코드 생성
실험 조건
UML 없이 생성
Problem Description만을 프롬프트로 사용해 AI 코드를 생성. 별도의 모델링 없이 텍스트 기반 명세만 사용.
비교 기준
Java vs Go
  • Java: 계층형 구조, 콘솔 중심으로 수렴
  • Go: 마이크로서비스, REST 중심으로 수렴
WS0의 핵심 발견
언어가 설계를 유도
언어와 프롬프트 차이가 AI의 아키텍처 선택에 영향을 준다. 즉, WS0는 언어 중심 비교였다.
W1의 핵심 질문: 명세 수준(UML 모델)을 높이면 AI는 더 구조적이고 검증 가능한 코드를 생성하는가?

L1 vs L2 코드 생성 비교
L1 — Use Case Diagram 기반
구조: 클래스/서비스 뼈대는 형성됨. Actor-UC 대응은 확인 가능.
조건 검증: 추상적이거나 일부 누락 가능성. Precondition이 코드에 약하게 반영.
예외 흐름: 만석·중복·노쇼 등 일부 단순화 또는 누락 가능성.
상태 전이: PENDING 등 상태가 생략되거나 단순화될 수 있음.
TODO 품질: 넓고 모호한 TODO. 구현 범위 불명확.
임의 추론: 명세에 없는 내용이 상대적으로 많이 추가될 수 있음.
VS
L2 — Use Case Description 기반
구조: 클래스/서비스 뼈대 + UC 단위 메서드 대응 명확.
조건 검증: Precondition 6개가 코드 레벨 검증 로직으로 반영될 가능성 큼.
예외 흐름: A1~A6 대안 흐름이 분기로 명확히 구현될 가능성 높음.
상태 전이: PENDING → CONFIRMED / CANCELLED 등 상태가 코드에 드러남.
TODO 품질: 좁고 구체적인 TODO. 미구현 지점이 명확함.
임의 추론: 명세 기반이므로 임의 추가 줄고, 필요 흐름 누락 감소.

Step 3 — L1 vs L2
기대 결과
명세 수준 상승의 기대 효과
구조적 명확성
뼈대는 L1에서도 충분
Use Case Diagram만으로도 Actor-UC 대응 구조는 생성된다. 그러나 L1과 L2의 차이는 구조 자체보다 세부 조건과 예외 처리에서 더 뚜렷하게 나타난다.
조건 검증 · 예외 흐름
L2에서 현저히 풍부
Precondition과 Alternative Flow가 명시된 L2에서는 검증 로직과 예외 분기가 훨씬 더 구체적으로 반영될 것으로 기대된다.
TODO 품질
목적 있는 미구현 지점
L2의 TODO는 명세에 근거한 미구현 지점을 더 정확히 가리킨다. L1의 광범위한 TODO와 달리 실행 가능한 작업 단위로 바뀐다.
종합 기대
명세 수준의 향상(L1 → L2)이 AI 생성 코드의 검증 가능성, 예외 처리, 상태 전이 표현력을 실제로 높이는지 관찰할 수 있다.

명세 수준의 단계적 상승
Problem Description → Use Case Diagram → Use Case Description으로 명세 수준을 단계적으로 높이며, AI 코드 생성 품질의 변화를 체계적으로 관찰했다.
DineSpot 적합성
비교 실험에 최적인 사례
Actor가 명확하고, 대안 흐름이 풍부하며, 상태 전이와 동시성 이슈가 존재해 명세 수준의 차이가 코드에 잘 드러나는 조건을 갖추고 있다.
Use Case Diagram 수준의 명세만으로도 AI는 시스템 구조의 윤곽을 생성할 수 있다.
그러나 Use Case Description 수준까지 구체화하면, 검증 조건, 대안 흐름, 상태 전이를 더 명확히 반영한 코드를 생성하는 경향이 뚜렷해진다.
